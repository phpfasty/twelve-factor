# Development

Front-end and template development: Latte templates, Tailwind CSS, design tokens, and how data is prepared for views.

---

## Templates (Latte)

### Layout and pages

- **Layout**: `templates/layout.latte` — global shell: `<html>`, `<head>`, `<body>`, header, main slot, footer. Every page is rendered inside this layout.
- **Pages**: `templates/pages/*.latte` — per-route content. The layout includes the current page output in `<main>` via the `content` variable.

The layout receives at least:

- `$title` — resolved page title (from `config/pages.php` title pattern).
- `$content` — HTML of the current page (rendered from the page template; use `{$content|noescape}`).
- `$site` — from the `site` data key (name, description, copyright, etc.).
- `$navigation` — from the `navigation` data key (e.g. `main` array for nav links).

So in `layout.latte` you use `$site`, `$navigation`, `$title`, and `$content`. Page templates receive all data keys defined for that route in `config/pages.php` (see [Preparing data for templates](#preparing-data-for-templates)).

### Referencing CSS

The layout links the compiled stylesheet:

```html
<link rel="stylesheet" href="/static/css/styles.css">
```

Only `styles.css` is used at runtime. The source for the build is `public/static/css/input.css` (see [Tailwind](#tailwind-css)).

### Latte basics

- Variables: `{$variable}`, `{$array['key']}`.
- Control structures: `{foreach $items as $item}...{/foreach}`, `{if}...{/if}`.
- Unescaped HTML: `{$content|noescape}` (used for the main content slot).
- Latte compiles templates to PHP under `templates/cache/`; that cache is separate from the app’s page cache in `cache/`.

## Tailwind CSS

### Input vs output

- **Source**: `public/static/css/input.css` — you edit this file. It imports Tailwind, defines design tokens, and can add `@layer base` / `@layer components` rules.
- **Output**: `public/static/css/styles.css` — generated by the Tailwind CLI. The app and layout reference only `styles.css`; do not edit it by hand.

### Build commands

From the **`public/`** directory (so paths match `package.json`):

```bash
cd public
npm install
npm run build:css     # one-off build: input.css → styles.css
npm run dev:css       # watch mode: rebuild on change
```

`package.json` scripts:

- `dev:css`: `npx @tailwindcss/cli -i ./static/css/input.css -o ./static/css/styles.css --watch`
- `build:css`: same without `--watch`

Run `npm run dev:css` while developing so CSS updates as you change `input.css` or templates.

### Scanning templates and PHP

Tailwind v4 needs to see where classes are used. In `input.css`:

```css
@import "tailwindcss";
@source "../../../templates/**/*.latte";
@source "../../*.php";
```

So Tailwind scans:

- All `.latte` files under `templates/` (relative to `public/static/css/` this is `../../../templates/`).
- PHP files in the project root (e.g. `config/`, `src/` if you use classes there). Adjust `@source` if your class usage lives elsewhere.

Adding a new template or moving files may require updating these `@source` paths so the built CSS includes the utilities you use.

## Design tokens

Design tokens live in `public/static/css/input.css`. They follow an HSL-based scheme (similar to shadcn) and are bridged into Tailwind v4 via `@theme inline`.

### Where they are defined

1. **`:root`** — light mode: colors (e.g. `--background`, `--foreground`, `--primary`, `--accent`), typography (`--font-display`, `--font-ui`, `--font-mono`), radius (`--radius-sm`, `--radius-md`, …), shadows. Values are raw HSL components (e.g. `42 26% 97%`) so they can be wrapped in `hsl()` in the bridge.
2. **`.dark`** — dark mode overrides for the same variables. Toggle dark mode by adding/removing the `dark` class on a parent (e.g. `<html class="dark">`).
3. **`@theme inline`** — maps those CSS variables into Tailwind’s theme so you can use `bg-background`, `text-foreground`, `rounded-md`, `font-display`, etc. For example:
   - `--color-background: hsl(var(--background));`
   - `--font-display: var(--font-display);`
   - `--radius-md: var(--radius-md);`

So the flow is: **tokens in `:root` / `.dark`** → **`@theme inline`** → **Tailwind utility classes**.

### Changing the look

- **Colors / typography / radius**: Edit the `:root` and `.dark` blocks in `input.css`. Keep the same variable names so `@theme inline` still works, or add new variables and corresponding entries in `@theme inline`.
- **New token**: Add a variable in `:root` (and `.dark` if needed), then add the mapping in `@theme inline` (e.g. `--color-*` for colors, `--font-*` for fonts). Rebuild CSS.
- **Base styles**: Use `@layer base` in `input.css` for global element styles (e.g. `body`, `a`, `h1`–`h6`). They can use the same tokens, e.g. `color: hsl(var(--foreground));`.

The layout currently uses a few Tailwind classes directly (e.g. `bg-slate-950 text-white`). For full token consistency you can switch those to token-based classes (e.g. `bg-background text-foreground`) once tokens are defined and bridged.

## Preparing data for templates

Templates receive one merged data array per request. The app builds it from config and the data provider; you don’t write PHP in templates to fetch data.

### 1. Page config (`config/pages.php`)

Each route defines:

- **`template`** — which file under `templates/pages/` to render (e.g. `about.latte`).
- **`title`** — string with optional placeholders, e.g. `{site.name} — About`. Placeholders use dot notation: `{site.name}`, `{post.title}`.
- **`data`** — list of data keys to load from the data provider (e.g. `['site', 'navigation', 'about']`). Each key corresponds to a fixture (e.g. `about.json`) or another source.
- **`dynamic`** (optional) — for routes like `/blog/@slug`: which dataset and collection to use, which field to match (e.g. `slug`), and under what key to expose the found item (e.g. `post`).

Example:

```php
'/about' => [
    'template' => 'about.latte',
    'title' => '{site.name} — About',
    'data' => ['site', 'navigation', 'about'],
],
'/blog/@slug' => [
    'template' => 'blog-post.latte',
    'title' => '{post.title} — {site.name}',
    'data' => ['site', 'navigation', 'blog'],
    'dynamic' => [
        'param' => 'slug',
        'dataset' => 'blog',
        'collection' => 'posts',
        'lookup' => 'slug',
        'item' => 'post',
    ],
],
```

### 2. Data provider and fixtures

- **Data keys** in `data` are resolved by `DataProviderInterface`. With `FixtureDataProvider`, key `about` → `fixtures/<locale>/about.json` for the active locale (e.g. `fixtures/en/about.json`), falling back to `fixtures/about.json` only if the localized file is missing.
- So: **fixture structure = template contract**. If `about.latte` uses `$about['hero']['title']`, then `fixtures/en/about.json` (for English) must have `hero.title`. Add or change keys in the fixture and use them in the template; no extra “preparation” layer besides the page config.

### 3. How the array is built (`PageRenderer`)

For each request, `PageRenderer::buildPageData()`:

1. Loads all datasets for the page’s `data` keys via `$dataProvider->getMany($dataKeys)` — e.g. `site`, `navigation`, `about` become top-level keys in the array.
2. If the route has `dynamic`, finds the matching item in the given dataset/collection by route parameter (e.g. slug) and adds it under the `item` key (e.g. `post`).
3. Merges route parameters (e.g. `slug`) into the array.
4. Resolves `title` by replacing `{...}` placeholders with values from that same array (dot notation, e.g. `site.name`), and sets `$pageData['title']`.

The result is passed to the page template and then to the layout (with `content` and `title`). So in a template you have:

- One object per `data` key: `$site`, `$navigation`, `$about`, etc.
- For dynamic pages: the resolved item, e.g. `$post`.
- `$title` already resolved (and again in layout).

### 4. Checklist for adding or changing a page

- **New page**: Add an entry in `config/pages.php` (path, `template`, `title`, `data`). Create the Latte file in `templates/pages/`. Add or reuse fixtures for each key in `data`; structure the JSON so the template can read it (e.g. `$about['hero']['title']`).
- **New data key**: Add the key to `data` for the route and add a fixture under the locale directory (e.g. `fixtures/en/newkey.json`). Use the key in the template as `$newkey`.
- **Dynamic page**: Add a route with `@param` in the path and a `dynamic` block (param, dataset, collection, lookup, item). Ensure the fixture has a collection (e.g. `posts`) and each item has the lookup field (e.g. `slug`). In the template use the `item` key (e.g. `$post`).
- **Title**: Use placeholders like `{site.name}` or `{post.title}` in `config/pages.php`; they are resolved from the same page data array.

No need to write PHP in templates or in a controller to “prepare” data — it’s all driven by `config/pages.php` and the fixture files.
